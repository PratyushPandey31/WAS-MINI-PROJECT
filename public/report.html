Chapter 1

Abstract & Project Objectives

1.1 Abstract

AuthGuard-MFA is a full-stack, production-grade Web Application Security capstone project that implements a Zero-Trust Identity & Access Management (IAM) system integrated with a live Security Information and Event Management (SIEM) console. The system demonstrates comprehensive compliance with the CIS Security Benchmarks, OWASP Top 10 vulnerability mitigations, and real-world Cloud / Virtualization Security architecture principles.

The project is built on Node.js with Express.js as the backend framework and a vanilla HTML5/CSS3 glassmorphic frontend. Core security primitives include HMAC-SHA256 signed JSON Web Tokens (JWT), bcrypt password hashing at work factor 12, Time-based One-Time Password (TOTP) Multi-Factor Authentication via the RFC 6238 standard, and HTTP Security Headers enforced through Helmet.js middleware.

The Audit SIEM Console provides real-time visibility into security events (AUTH_SUCCESS, AUTH_FAILED, BRUTE_FORCE_SUSPECTED) streamed from the SQLite event log. Role-Based Access Control (RBAC) governs data visibility across three access tiers: Administrator, Security Operations (SecOps), and Guest. A suite of interactive exploit simulators — including Brute-Force Attack, XSS Sanitization, JWT Tampering, and Session Cookie Auditor — allows live demonstration of attack vectors and corresponding defensive countermeasures.

1.2 Project Objectives

  1. Design and implement a Zero-Trust authentication pipeline using JWT + TOTP MFA.
  2. Enforce CIS Benchmark controls (4.1, 5.2, 6.3, 8.1, 9.4) in a deployed application.
  3. Build a real-time SIEM console for security event ingestion, parsing, and visualization.
  4. Demonstrate OWASP Top 10 mitigations (A01–A07) through live exploit simulators.
  5. Design a Cloud VPC architecture following network isolation and Zero-Trust principles.
  6. Implement Role-Based Access Control (RBAC) for multi-tier data governance.
  7. Enable export of security compliance reports in Excel (CSV) and JSON format.

1.3 Scope & Significance

The project addresses the growing industry demand for security-by-design software development. According to the Verizon 2024 Data Breach Investigations Report, 61% of breaches involve stolen credentials. This project provides a tangible, working demonstration of credential-hardening techniques aligned with industry frameworks such as NIST SP 800-63B, CIS Controls v8, and OWASP Application Security Verification Standard (ASVS) Level 2.

2.3 Request Authentication Flow

Every protected route follows this Zero-Trust verification pipeline:

Client → POST /api/auth/login {username, password} Server → 1. Lookup user in SQLite (parameterized query) → 2. bcrypt.compare(password, stored_hash) → 3. If MFA enabled: issue pre-auth token (mfa_verified=false) → 4. Client submits TOTP OTP → speakeasy.verify() → 5. Issue full-access JWT (HMAC-SHA256, exp: 15m) → 6. Log AUTH_SUCCESS event to SIEM with IP, timestamp Protected API: JWT middleware verifies signature + expiry on every request

3.3 JWT Cryptographic Security Analysis

The system uses HMAC-SHA256 (HS256) for JWT signing. The token structure is:

Header: { "alg": "HS256", "typ": "JWT" } Payload: { "id": 1, "username": "admin", "mfa_enabled": true, "mfa_verified": true, "iat": 1719558000, "exp": 1719558900 } Signature: HMAC-SHA256(base64(header) + "." + base64(payload), JWT_SECRET)
⚠️ Key Entropy Requirement (OWASP): JWT secrets must be a minimum of 256 bits (32+ high-entropy characters). The Cryptographic Key Entropy Auditor built into this project grades any JWT secret on a Shannon Entropy scale from A+ (≥4.5 bits/char) to F (<2.0 bits/char).

Chapter 4

System Modules & Feature Description

4.1 Authentication & Identity Module

  • User Registration: Collects username, email, and password. Password hashed with bcrypt (WF-12) before insertion into SQLite. Duplicate detection prevents account takeovers.
  • Login — Step 1: Username/password verified. If MFA disabled, full-access JWT issued immediately. If MFA enabled, a pre-auth JWT (mfa_verified=false) is issued, prompting OTP entry.
  • TOTP MFA — Step 2: 6-digit RFC 6238 TOTP verified by speakeasy with ±1 step tolerance (30-second windows). On success, a full-access JWT is issued.
  • QR Code Enrollment: MFA setup page generates a base32 TOTP secret and renders a QR code using the qrcode library compatible with Google Authenticator and Authy.
  • JWT Expiry & Refresh: All tokens expire in 15 minutes. Dashboard auto-detects expiry and redirects to login — enforcing session security.

4.2 Cloud Console Dashboard

  • Welcome Greeting: Personalized "Welcome back, [username] 👋" with live clock (Good Morning / Afternoon / Evening) based on local time.
  • Metric Stat Cards: Four KPI cards — Security Status (SECURE), MFA Status (ON ✓ / OFF), CIS Controls Score (5/5), Cloud Platform (AWS).
  • CIS Compliance Auditor: Five live compliance rows with interactive hover effects. MFA row updates dynamically from JWT claims.
  • JWT Token Decoder: Live base64-decode of the active token showing Header (red), Payload (cyan), and HMAC-SHA256 Signature (green).
  • Token Tamper Playground: Edit the active JWT, submit it — the server's HMAC validation returns HTTP 401 Unauthorized, demonstrating tamper-proofing.
  • Entropy Auditor: Shannon Entropy calculator grades JWT secret key strength from A+ to F with actionable recommendations.
  • Security Report Export: Downloads a full compliance posture report as .csv (Excel-compatible with UTF-8 BOM) or .json.

4.3 Audit SIEM Console

  • Live Security Event Feed: Polls /api/auth/logs and renders structured JSON events in a terminal-style table with color-coded severity badges (INFO / WARNING / CRITICAL).
  • RBAC Role Selector: Admin (full logs + hashes), SecOps (redacted hashes), Guest (access denied). Demonstrates principle of least privilege.
  • Brute-Force Simulator: Fires 10 rapid POST /login requests. Rate limiter triggers HTTP 429. SIEM captures BRUTE_FORCE_SUSPECTED CRITICAL events live.
  • XSS Sanitizer Demo: Input like <script>alert()</script> is escaped to HTML entities on the backend — output is safe and non-executable.
  • Session Cookie Auditor: Proves HttpOnly cookies cannot be accessed via document.cookie in JavaScript — XSS cookie theft mitigation confirmed.
  • Log Ingest & Parse: Regex-based parser extracts timestamp, event type, IP, and severity from raw log strings. CRITICAL events simulate an AWS SNS alarm payload.
  • SQLite DB Inspector: Renders the full users table. Passwords are shown as bcrypt hashes (redacted to [REDACTED] in SecOps role).
  • SIEM Export: Downloads all visible security events to .csv with date, title, and proper column headers.
✅ Live Demo Proof of Concept: All features are functional in the running application at http://localhost:3000. The brute-force simulator, JWT tamper playground, and XSS sanitizer produce real server-side responses visible in the SIEM event log — not simulated client-side fakes.

5.3 Conclusion

AuthGuard-MFA successfully demonstrates a production-ready, security-by-design web application that integrates multiple industry-standard security controls into a cohesive, functional system. The project bridges theoretical knowledge of CIS Benchmarks, OWASP standards, and Zero-Trust Architecture with practical, working implementation.
